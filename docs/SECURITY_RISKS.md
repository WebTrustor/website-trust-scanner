|---------|--------|

## 1. مخاطر أمنية تقنية

### 🔴 SSRF (Server-Side Request Forgery)
**الخطر:** المنصة تطلب URLs يُدخلها المستخدم. بدون validation صارم، يمكن لمهاجم إرسال طلبات إلى:
- البنية التحتية الداخلية
- cloud metadata endpoints (169.254.169.254)
- خدمات داخلية على الشبكة الخاصة
- ملفات محلية (file://)

**الإجراء:** URL Safety Validator إلزامي في المرحلة 3 قبل أي فحص.  
**الاختبارات المطلوبة:** اختبار شامل لجميع حالات SSRF.

---

### 🔴 تسريب API Keys
**الخطر:** Google Safe Browsing، VirusTotal، وغيرها تتطلب مفاتيح API. تسريبها يعني:
- استنزاف الحصة (quota)
- فاتورة ضخمة
- إساءة استخدام خدمات المنصة

**الإجراء:**
- تخزين جميع المفاتيح في environment variables فقط
- لا تظهر في الكود، الـ logs، الـ responses، أو الـ error messages
- تشفيرها في قاعدة البيانات إذا خُزنت هناك
- استخدام secrets manager في الـ production

---

### 🔴 Authorization Bypass
**الخطر:** إذا كان التحقق من الصلاحيات في Frontend فقط، يمكن لمهاجم استدعاء API endpoints مباشرة وتشغيل Deep Scan بدون إذن.

**الإجراء:**
- جميع قواعد الصلاحيات تُطبَّق في Backend
- كل endpoint يتحقق من: الدور + حالة الموقع + Authorization Record
- الـ Frontend validation هو UI/UX فقط، وليس أمانًا

---

### 🔴 Scan Policy Bypass
**الخطر:** إذا كانت سياسة الفحص قواعد "مكتوبة" وليست مُطبَّقة برمجيًا، يمكن تجاوزها.

**الإجراء:**
- بناء Scan Policy Engine مستقل في المرحلة 3
- كل طلب فحص يمر عبر Policy Engine أولًا
- الـ Policy Engine يتحقق من حالة الموقع + Do Not Scan List + Authorization Record

---

### 🟠 Insecure Direct Object Reference (IDOR)
**الخطر:** إذا كانت scan IDs قابلة للتخمين (1, 2, 3...)، يمكن لمستخدم الوصول إلى تقارير مستخدمين آخرين.

**الإجراء:**
- UUID v4 لجميع الـ IDs الحساسة
- التحقق من ملكية المورد في كل طلب
- تقارير العامة لها رابط مؤقت (24 ساعة)

---

### 🟠 Data Exfiltration عبر Scan Results
**الخطر:** إذا عرضت تقارير الفحص الحساس معلومات دقيقة جدًا (مسارات حساسة، معلومات server)، يمكن للمهاجم استخدام المنصة كأداة استطلاع.

**الإجراء:**
- Trust Report (Public) لا يعرض أي معلومات قابلة للاستغلال
- Lead Audit Report لا يعرض مسارات أو headers حساسة
- كل نوع تقرير له template محدد مسبقًا

---

### 🟠 Rate Limiting Abuse
**الخطر:** بدون rate limiting، يمكن لمهاجم استخدام المنصة لـ:
- فحص آلاف المواقع (DDoS من الدرجة الثانية)
- استنزاف حصص API الخارجية
- تحميل زائد على الخوادم

**الإجراء:**
- Rate limiting على مستوى IP (Guests)
- Rate limiting على مستوى Account (Users)
- Queue للمهام الطويلة
- Timeout صارم لكل فحص

---

### 🟠 Scan as DDoS Vector
**الخطر:** يمكن لمستخدم خبيث إدخال URL لموقع ضحية وتشغيل فحوصات متكررة، مما يشكّل DDoS غير مباشر.

**الإجراء:**
- Rate limiting صارم لكل domain target
- الحد الأقصى لعدد الطلبات لنفس الـ domain في 24 ساعة
- Do Not Scan List للمواقع الحساسة
- منع فحص نفس الـ domain أكثر من X مرة بدون تأخير

---

### 🟡 PDF Report Injection
**الخطر:** إذا كانت البيانات المُدخلة من الموقع المفحوص تُضمَّن في PDF بدون تنقية، قد يحتوي PDF على محتوى ضار.

**الإجراء:**
- تنقية جميع البيانات قبل إدخالها في قوالب PDF
- استخدام template engine آمن
- لا تدمج HTML/JS مباشرة من الموقع المفحوص في التقرير

---

### 🟡 Log Injection
**الخطر:** إذا سُجّلت URLs أو headers كما هي، يمكن حقن بيانات في الـ logs.

**الإجراء:**
- تنقية جميع البيانات الخارجية قبل تسجيلها
- لا تُسجّل قيم headers الكاملة بدون تحقق
- لا تُسجّل API Keys أو tokens

---

## 2. مخاطر قانونية وأخلاقية

### 🔴 الفحص بدون إذن
**الخطر:** تنفيذ فحوصات عميقة على مواقع بدون موافقة صاحبها قد يُعتبر:
- جريمة في قوانين الجرائم الإلكترونية (مثل CFAA في أمريكا، وقوانين مشابهة في السعودية والإمارات ومصر)
- انتهاكًا لـ GDPR
- قد يُعرّض المنصة لدعاوى قضائية

**الإجراء:**
- الفصل الصارم بين Public Check (سطحي) وDeep Scan (بعد إذن)
- توثيق كل Authorization Record
- شروط الخدمة تنص بوضوح على حدود الاستخدام
- منع أي فحص عميق برمجيًا بدون Authorization Record صالح

---

### 🔴 إساءة الاستخدام التسويقي (Lead Audit)
**الخطر:** إذا استخدم Admin ميزة Lead Audit بشكل مفرط أو لأغراض استطلاع، قد يُعتبر هذا:
- جمع معلومات غير مصرح
- انتهاكًا لـ GDPR (خاصة للمواقع الأوروبية)
- تدخلًا غير قانوني

**الإجراء:**
- Lead Audit محدود ببيانات عامة فقط
- Rate limiting صارم على Lead Audit
- Audit Log لكل عملية Lead Audit
- Terms of Service تحدد الاستخدام المقبول

---

### 🟠 تخزين البيانات الشخصية
**الخطر:** إذا خُزّنت بيانات شخصية (IP، بريد إلكتروني) بدون سياسة واضحة، يُشكّل ذلك انتهاكًا لـ GDPR وقوانين الخصوصية.

**الإجراء:**
- تحديد ما يُخزَّن وما لا يُخزَّن
- سياسة خصوصية واضحة
- آلية حذف البيانات بطلب المستخدم
- تحديد مدة الاحتفاظ لكل نوع بيانات

---

### 🟠 Outreach Report كأداة تشهير
**الخطر:** إذا شارك Admin تقرير Lead مع جهة ثالثة وهو يحتوي معلومات مسيئة عن موقع، قد يُشكّل ذلك مشكلة قانونية.

**الإجراء:**
- Outreach Report يجب أن يكون مهنيًا ومحايدًا
- لا يُضمَّن أي معلومات قابلة للاستغلال
- مراجعة قالب الـ Outreach Report قبل الإطلاق

---

### 🟡 Liability عند اكتشاف ثغرة
**الخطر:** إذا اكتشفت المنصة ثغرة خطيرة في موقع عميل وأبلغته، لكن لم يُصلحها وتعرض للاختراق، هل المنصة مسؤولة؟

**الإجراء:**
- توضيح في Terms of Service أن التقارير إرشادية وليست ضمانات
- تحديد مسؤولية الإصلاح على صاحب الموقع
- توثيق إرسال التقرير (Audit Log)

---
