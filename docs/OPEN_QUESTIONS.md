## القسم الأول: قرارات تقنية جوهرية

### Q1: Backend Language / Framework
**الخيار A:** FastAPI (Python)  
- مزايا: سريع في بناء APIs، مكتبات أمن سيبراني ممتازة (dnspython, ssl, cryptography)، سهل للـ Workers بـ Celery
- عيوب: إذا كانت خلفيتك TypeScript فقط، تعلّم إضافي

**الخيار B:** NestJS (TypeScript/Node.js)  
- مزايا: نفس اللغة مع Next.js، TypeScript end-to-end
- عيوب: مكتبات الأمن أقل نضجًا من Python، Workers بـ BullMQ

**الخيار C:** FastAPI للـ Scanning + NestJS للـ API Gateway  
- مزايا: أفضل ما في الخيارين
- عيوب: تعقيد أعلى في MVP

**سؤالي لك:** ما هو تفضيلك التقني؟ هل لديك خلفية Python أم Node.js؟

### Q2: Reputation API الأساسي
للـ MVP يكفي provider واحد. الخيارات:

**الخيار A:** Google Safe Browsing API (مجاني بحدود)  
- مزايا: مجاني للاستخدام المعقول، موثوق
- عيوب: يحتاج Google API Key

**الخيار B:** VirusTotal API (مجاني بحدود)  
- مزايا: أشمل، يشمل malware + phishing
- عيوب: 4 طلبات/دقيقة في المجاني

**الخيار C:** Mock Provider في MVP (بدون API حقيقية)  
- مزايا: لا يحتاج keys، أسرع في التطوير
- عيوب: النتائج وهمية، لا قيمة حقيقية للمنتج

**الخيار D:** URLhaus + OpenPhish (مجاني تمامًا)  
- مزايا: مجاني بالكامل، لا API key
- عيوب: أقل شمولًا من Google/VirusTotal

**سؤالي لك:** هل لديك API keys جاهزة؟ أم نبدأ بـ Mock ثم نضيف؟

## القسم الثاني: قرارات منتج جوهرية

### Q4: نموذج الأعمال (Monetization)
هذا يؤثر على التصميم المعماري الآن:

**الخيار A:** Freemium  
- Public Check مجاني للجميع
- المراقبة الدورية وتقارير PDF بخطط مدفوعة
- نتيجة: يجب بناء نظام subscriptions لاحقًا

**الخيار B:** للشركات فقط (B2B)  
- المنصة للوكالات والشركات فقط
- لا نسخة مجانية للعامة
- نتيجة: Admin-first, simpler monetization

**الخيار C:** Hybrid  
- Public Check مجاني دائمًا
- Lead Audit للوكالات (باشتراك شهري)
- Owner Monitoring (باشتراك حسب عدد المواقع)

**سؤالي لك:** ما هو نموذجك التجاري المستهدف؟

### Q10: فترة صلاحية تقارير Public Check
